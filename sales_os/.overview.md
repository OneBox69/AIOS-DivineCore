# Sales OS — Module 2 (Revenue System)

Manages the full sales pipeline AND does active outbound. This is not a passive CRM — it initiates contact, sends messages, runs follow-up sequences, and (eventually) makes outbound calls.

## Status

**Planning / early build for the agent layer.** Hormozi knowledge base is identified but not yet ingested. No live AI agents yet.

**Live**:
- `integrations/upwork/` + `web/upwork_routes.py` — proposal generator (mounted at `/upwork`)
- `integrations/upwork_jobs/` + `web/upwork_jobs_routes.py` — daily Apify scrape into manual review queue (mounted at `/upwork-jobs`)
- `integrations/instantly/` + `web/instantly_routes.py` — cold-email outreach (webhook + dashboard at `/outreach`)

**Scaffolding (not yet wired into the divinecore-v2 runtime)**:
- `Clients/icps/` — per-client ideal-customer-profile YAML configs (ecommerce in)
- `framework/` — reusable templates for new-client onboarding (qualify, scrape, email, etc.)
- `workflows/{copy,export,n8n,scrape}/` — Python scripts + n8n JSON exports for outbound pipeline. Reference / WIP — these are not Celery tasks today.

## Agents (planned)

| Agent | Role |
|-------|------|
| Lead Agent | Stores leads, categorises hot/warm/cold, manages CRM state |
| Follow-up Agent | Generates personalised follow-ups, drives nurture sequences |
| Deal Agent | Tracks deals through Lead → Contacted → Interested → Closed; handles client memory |

## Knowledge base

**Source**: Alex Hormozi's $100M series — `$100M Offers` and `$100M Leads`. Full frameworks for offer construction, money models, lead conversion, value equation.

This is the domain expertise layer. Applied directly inside the agents' reasoning — system prompts will reference Hormozi frameworks the way Branding OS references Kallaway.

Storage decision (TBD when build starts): Airtable for structured frameworks (mirroring the Branding OS pattern), possibly Supabase for conversation/example retrieval.

## Capabilities (planned)

- Lead storage + categorisation (hot/warm/cold)
- AI-generated personalised follow-ups
- Pipeline tracking: Lead → Contacted → Interested → Closed
- Conversation memory (last message, next action, deal state)
- Outbound channels: DMs, LinkedIn, WhatsApp, calls (when budget allows)

## Context sources (REQUIRED — every Sales OS agent loads these)

Sales OS produces outreach copy and conversation responses to real prospects. Every agent here MUST pull identity context from `shared/context/` in addition to the Hormozi KB:

| File | Why it matters for Sales OS |
|------|-----------------------------|
| [`shared/context/identity/business-info.md`](../shared/context/identity/business-info.md) | What we sell, what we don't, the AI OS vs AI tools framing. The buyer-facing pitch. |
| [`shared/context/identity/voice.md`](../shared/context/identity/voice.md) | Tone — direct, operator-to-operator. Email length conventions. Banned phrases that would scream "consultant." |
| [`shared/context/identity/audience.md`](../shared/context/identity/audience.md) | Reader profile + sub-segments (agency owner, eCom founder, solo operator). Determines which pain points to lead with. |
| [`shared/context/identity/strategy.md`](../shared/context/identity/strategy.md) | Current monetisation layers — what we should and shouldn't pitch right now (e.g. don't pitch coaching yet). |

Pattern: agent system prompts should cite the relevant `shared/context/` files explicitly (e.g. *"Compose the email using the tone described in `shared/context/identity/voice.md` for the audience profile in `shared/context/identity/audience.md`"*).

Note the split with Branding OS: Branding OS loads team-member persona files (`mayank.md` / `pang.md`) for first-person content, Sales OS loads `business-info.md` for third-person buyer-facing pitch. Different reader, different framing. (When sales emails go out under a specific person's name, additionally load that person's persona file.)

## What goes here

- Agent system prompts → `agents/` (one file per agent)
- n8n workflow JSON exports → `workflows/`
- Hormozi KB sync scripts → (when added) `knowledge-base/`
- External-system Python integrations (Celery tasks + helpers) → `integrations/<vendor>/`
- FastAPI routers + Jinja templates that render Sales OS UIs → `web/`

## What does NOT go here

- Lead data itself — that's in Airtable / CRM, not files.
- Sent message logs — those are in Airtable / Discord / wherever the channel lives.
- Brand voice or buyer personas — those live in `shared/context/`. Don't duplicate.
