# sales_os/integrations/instantly/

Cold-email outreach pipeline. Webhook ingest of replies + daily campaign analytics snapshot to Supabase. Real-time Discord ping for positive replies. Mirrors the `ops_os/integrations/fathom/` shape.

## Flow

```
Instantly webhook (POST /instantly/webhook)
  -> sales_os/web/instantly_routes.py validates X-Webhook-Secret
  -> tasks.process_instantly_reply.delay(payload)         # processor.py
       -> categories.is_ignored?  -> drop (OOO / auto-reply)
       -> supabase_writer.insert_reply(...)               # outreach_replies row
       -> categories.is_positive? -> discord_notify.post_positive_reply
                                       (thread inside #sales-and-outreach)

Beat (daily 00:30 UTC)
  -> tasks.poll_instantly_campaigns                       # processor.py
       -> client.list_campaigns()
       -> for each: client.get_campaign_analytics + get_campaign_steps_analytics
       -> supabase_writer.upsert_campaign (positioning preserved via ?columns=)
       -> supabase_writer.upsert_sequence_step  (one row per step)
       -> supabase_writer.upsert_daily_metrics  (one row per (campaign, step, yesterday))
```

## Files

| File | Role |
|------|------|
| `client.py` | httpx wrapper around Instantly's v2 REST API. Endpoint paths verified against live account on first poll — raw responses also stored in `outreach_daily_step_metrics.raw_snapshot` for forensics if shapes change. |
| `categories.py` | `POSITIVE_CATEGORIES`, `IGNORED_CATEGORIES`. Case-insensitive matching. Adjustable as we observe Instantly's exact label vocabulary. |
| `supabase_writer.py` | Upserts for the four `outreach_*` tables + read helpers used by the dashboard. PostgREST + httpx, same auth pattern as `ops_os/integrations/fathom/supabase_writer.py:12`. |
| `discord_notify.py` | `post_positive_reply()` — single embed POST. Targets the thread inside `#sales-and-outreach` via `?thread_id=` on the webhook URL. No-op if either env var is unset. |
| `processor.py` | Two Celery tasks: `process_instantly_reply` (webhook) and `poll_instantly_campaigns` (beat). |

## Supabase schema

Run once in the Supabase SQL editor before deploying:

```sql
create table if not exists public.outreach_campaigns (
    id           text primary key,                  -- Instantly campaign id
    name         text not null,
    positioning  text not null default '',          -- "UK ecom beauty 5-50 employees"
    status       text,
    started_at   timestamptz,
    created_at   timestamptz not null default now(),
    updated_at   timestamptz not null default now()
);

create table if not exists public.outreach_sequence_steps (
    id            uuid primary key default gen_random_uuid(),
    campaign_id   text references public.outreach_campaigns(id) on delete cascade,
    step_index    int  not null,
    subject       text,
    body_preview  text,
    unique (campaign_id, step_index)
);

create table if not exists public.outreach_daily_step_metrics (
    id              uuid primary key default gen_random_uuid(),
    campaign_id     text references public.outreach_campaigns(id) on delete cascade,
    step_index      int  not null,
    date            date not null,
    emails_sent     int  not null default 0,
    replies         int  not null default 0,
    unsubscribes    int  not null default 0,
    raw_snapshot    jsonb,
    unique (campaign_id, step_index, date)
);
create index if not exists outreach_metrics_date_idx
    on public.outreach_daily_step_metrics (date desc);

-- emails_opened and bounces were dropped in commit  — opens are typically
-- 0 on cold email without pixel tracking and bounces weren't in the
-- analytics endpoint's payload. raw_snapshot still preserves the original
-- response if either becomes useful later.

create table if not exists public.outreach_replies (
    id                  uuid primary key default gen_random_uuid(),
    instantly_lead_id   text,
    lead_email          text not null,
    lead_company        text,
    campaign_id         text references public.outreach_campaigns(id) on delete set null,
    step_index          int,
    replied_at          timestamptz not null,
    body                text,
    instantly_category  text,
    is_positive         boolean not null default false,
    reply_url           text,
    raw_payload         jsonb,
    created_at          timestamptz not null default now()
);
create index if not exists outreach_replies_campaign_idx
    on public.outreach_replies (campaign_id, replied_at desc);
create index if not exists outreach_replies_category_idx
    on public.outreach_replies (instantly_category);

alter table public.outreach_campaigns enable row level security;
alter table public.outreach_sequence_steps enable row level security;
alter table public.outreach_daily_step_metrics enable row level security;
alter table public.outreach_replies enable row level security;
-- No policies — worker uses SUPABASE_SECRET_KEY which bypasses RLS.
```

## Positioning is operator-owned

The poller never writes `outreach_campaigns.positioning`. It defaults to `''` on insert. After the first poll lands a campaign row, set its positioning manually in Supabase Studio:

```sql
update public.outreach_campaigns
set positioning = 'UK ecom beauty 5-50 employees'
where id = '<instantly_campaign_id>';
```

The dashboard renders `(unset)` for empty positioning so it's visible when missing.

## Env vars

Read from `worker/settings.py`:
- `INSTANTLY_API_KEY` — Bearer token from Instantly settings
- `INSTANTLY_API_BASE_URL` — default `https://api.instantly.ai`
- `DISCORD_OUTREACH_WEBHOOK_URL` — webhook on `#sales-and-outreach`
- `DISCORD_OUTREACH_THREAD_ID` — thread id of the `outreach-replies` thread inside that channel
- `SUPABASE_URL`, `SUPABASE_SECRET_KEY` — already configured (shared with Fathom)

Read from `api/settings.py` (FastAPI process):
- `INSTANTLY_WEBHOOK_SECRET` — shared secret Instantly sends in `X-Webhook-Secret` header

## Discord thread setup (one-time)

1. Create a webhook on `#sales-and-outreach` (Channel Settings -> Integrations -> Webhooks). Save its URL as `DISCORD_OUTREACH_WEBHOOK_URL`.
2. Manually create a thread named `outreach-replies` inside `#sales-and-outreach`. Right-click thread -> Copy ID. Save as `DISCORD_OUTREACH_THREAD_ID`.
3. Set the thread's auto-archive duration to the longest available option. Webhook posts to an archived thread silently un-archive it, but keeping it active is tidier.

## Public webhook URL

Production: `https://upwork.srv1445995.hstgr.cloud/instantly/webhook`

Routed through n8n's Traefik via a separate router (`instantly-webhook` priority 100, no basicauth middleware) so Instantly can POST without credentials. Auth happens inside the FastAPI handler via the `X-Webhook-Secret` header. Register this URL + secret in Instantly's webhook settings.

## Reply-payload tolerance

Webhook payload shapes vary by Instantly plan. `processor._extract_reply` reads from several common field names and falls back to empty values. The full raw payload is always stored in `outreach_replies.raw_payload` so you can inspect new field names in Supabase Studio and update extraction rules if Instantly changes shape.

## Manual trigger / debugging

Force a daily snapshot run:
```bash
docker compose exec worker celery -A celery_app call tasks.poll_instantly_campaigns
```

Replay a webhook locally:
```bash
curl -X POST http://localhost:8000/instantly/webhook \
  -H 'X-Webhook-Secret: <secret>' \
  -H 'Content-Type: application/json' \
  -d @sample_reply.json
```
