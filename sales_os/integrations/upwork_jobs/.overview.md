# sales_os/integrations/upwork_jobs/

Daily Apify-driven scraper for Upwork job listings. Pulls a single saved-search URL once a day at **05:40 MYT (= 21:40 UTC)**, upserts new jobs to Supabase, and exposes a manual review queue at `GET /upwork-jobs`. Pang reads through the queue, clicks **Generate proposal** on jobs worth applying to — that handoff invokes the existing `sales_os/integrations/upwork/` pipeline directly with the stored job description.

**No auto-drafting. No scoring. No Discord posting.** OpenRouter / Google credits are only burned when Pang explicitly clicks Generate on a specific job.

## Why Apify (and not the official Upwork API)

The official Upwork GraphQL API requires a partner application that takes weeks. Direct scraping of upwork.com gets accounts banned. Apify's `neatrat/upwork-job-scraper` actor solves the middle ground: it rotates *Apify's* proxies and cookies, so the ToS / detection risk lives on Apify's side rather than on Pang's account. Cost is ~$3-10/mo at one-run-per-day volume.

When the official API access lands later, replace `apify_client.py` with a GraphQL client and keep everything downstream unchanged.

## Flow

```
Beat (daily 21:40 UTC = 05:40 MYT)
  -> tasks.poll_upwork_jobs                    # processor.py
       -> apify_client.fetch_upwork_jobs(UPWORK_SEARCH_URL)
            -> POST https://api.apify.com/v2/acts/neatrat~upwork-job-scraper/run-sync-get-dataset-items
       -> for each item:
            -> _normalise(item)                # raw Apify shape -> our columns
            -> supabase_writer.upsert_job(...) # status / reviewed_at / applied_at preserved on conflict

Operator (any time, via /upwork-jobs UI)
  -> GET  /upwork-jobs?tab=new|skipped|applied  -> queue list
  -> GET  /upwork-jobs/{job_id}                 -> single-job detail
  -> POST /upwork-jobs/{id}/skip                -> status='skipped'
  -> POST /upwork-jobs/{id}/unskip              -> status='new'
  -> POST /upwork-jobs/{id}/unmark-applied      -> status='new'
  -> POST /upwork-jobs/{id}/generate
        -> _celery.send_task("tasks.upwork_generate_proposal", [stored_description])
        -> blocks on result (~60-120s)
        -> on success: status='applied' + applied_at=now() + proposal_doc_url=<url>
        -> renders the same upwork_result.html the /upwork form uses
```

## Files

| File | Role |
|------|------|
| `apify_client.py` | httpx wrapper around `POST /v2/acts/{id}/run-sync-get-dataset-items`. One call kicks off the run and returns scraped items. 5-minute timeout. |
| `supabase_writer.py` | `upsert_job` (operator-owned columns excluded via `?columns=`), `fetch_jobs_by_status`, `count_jobs_by_status`, `fetch_job`, `update_status`. Same auth + `Prefer: resolution=merge-duplicates,return=representation` pattern as `instantly/supabase_writer.py`. |
| `processor.py` | One Celery task: `poll_upwork_jobs`. Normalises Apify items and upserts. No-ops cleanly if `APIFY_API_TOKEN` or `UPWORK_SEARCH_URL` is unset. |

## Supabase schema

Run once in the Supabase SQL editor before deploying:

```sql
create table if not exists public.upwork_jobs (
    job_id                text primary key,           -- Apify item id (`id` / `jobId` / `ciphertext`)
    url                   text not null,
    title                 text not null,
    description           text not null,
    budget                jsonb not null default '{}'::jsonb,
    client_stats          jsonb not null default '{}'::jsonb,
    skills                text[] not null default '{}',
    posted_at             timestamptz,
    tags                  text[] not null default '{}',
    contract_type         text,
    engagement            text,
    country_restrictions  text[],                     -- null = open to all; [] = explicitly empty
    experience_level      text,
    category              text,
    raw                   jsonb not null,             -- full Apify item, for forensics
    fetched_at            timestamptz not null default now(),

    -- Operator-owned (set via the /upwork-jobs UI; the poller never touches these).
    status                text not null default 'new'
                          check (status in ('new', 'skipped', 'applied')),
    reviewed_at           timestamptz,
    applied_at            timestamptz,
    proposal_doc_url      text
);

create index if not exists upwork_jobs_status_posted_idx
    on public.upwork_jobs (status, posted_at desc nulls last);
create index if not exists upwork_jobs_fetched_idx
    on public.upwork_jobs (fetched_at desc);

alter table public.upwork_jobs enable row level security;
-- No policies — worker uses SUPABASE_SECRET_KEY which bypasses RLS.
```

## Operator-owned columns

`status`, `reviewed_at`, `applied_at`, `proposal_doc_url` are owned by the queue UI, never by the daily poller. The poller's upsert uses PostgREST's `?columns=...` to whitelist only the fields it sends — so a job that's been skipped (or applied) yesterday and reappears in today's scrape keeps its review state intact. Description, client stats, etc. **are** refreshed.

## Tabs (the queue UI)

- **New** (default) — `status='new'`, sorted by `posted_at desc`. Newest job posted on Upwork shows up first. Carries over indefinitely until you skip or apply.
- **Skipped** — `status='skipped'`. Includes an "Move back to New" button.
- **Applied** — `status='applied'`. Logs the proposal Doc URL the pipeline produced. Includes an "Unmark applied" button for accidental clicks.

Nothing auto-deletes. If the table ever gets uncomfortably large, add a manual purge SQL or a Celery beat task on a 30-day cadence.

## Apify setup (one-time)

1. Sign up at apify.com. Settings -> Integrations -> API token. Copy.
2. Add to `.env`: `APIFY_API_TOKEN=apify_api_...`.
3. The actor is `neatrat/upwork-job-scraper`. Default in settings — override with `APIFY_UPWORK_ACTOR_ID` if Apify ever migrates the actor.
4. Pricing on the actor: `$3.20 / 1,000 jobs`, minimum 10-job bill per run. One run/day with ~30-100 jobs/run = **~$3-10/mo**. Free tier (10 runs / 100 results total) is enough to test for one evening.

## The Upwork search URL

`UPWORK_SEARCH_URL` is whatever URL Pang pastes from upwork.com after configuring filters in Upwork's UI (recency, experience level, budget floor, location, etc.). Per the actor's docs, `rawUrl` overrides all other input parameters — so all filtering is controlled via the URL itself. Bake the recency filter into the URL (`&t=24` or similar) since we override nothing.

## Env vars

Read from `worker/settings.py`:
- `APIFY_API_TOKEN` — Bearer token from Apify
- `APIFY_UPWORK_ACTOR_ID` — default `neatrat/upwork-job-scraper`
- `UPWORK_SEARCH_URL` — Upwork search URL to scrape; empty = poller no-ops
- `SUPABASE_URL`, `SUPABASE_SECRET_KEY` — already configured (shared with Fathom + Instantly)

Read from `api/settings.py` (FastAPI process):
- `SUPABASE_URL`, `SUPABASE_SECRET_KEY` — read by the queue UI for list / detail / status updates

## Manual trigger / debugging

Force the daily poll outside of beat:
```bash
docker compose exec worker celery -A celery_app call tasks.poll_upwork_jobs
```

Inspect what Apify returned for a single job (use the `raw` jsonb column):
```sql
select job_id, raw -> 'allowedApplicantCountries', raw -> 'budget'
from public.upwork_jobs
where job_id = '<id>';
```

Reset a job's review state from the SQL editor (rare):
```sql
update public.upwork_jobs
set status = 'new', reviewed_at = null, applied_at = null, proposal_doc_url = null
where job_id = '<id>';
```

## Open questions

1. The actor's exact field names — `id` vs `jobId` vs `ciphertext`, `postedOn` vs `publishedOn`, `client` vs `clientStats` — are best-effort fallbacks. After the first real run, inspect a few `raw` rows in Supabase Studio and tighten `_normalise` if any field is consistently empty when it shouldn't be.
2. Pagination — `resultsPerPage=50` covers a typical day. If a daily run regularly returns 50 with more pending, switch to multi-page polling in `apify_client.fetch_upwork_jobs`.
3. Country restrictions UX — `country_restrictions = null` means "open to all", `[]` means "explicitly empty" (likely a scrape glitch). The list view should render "open to all" for null and the country list for non-empty.
