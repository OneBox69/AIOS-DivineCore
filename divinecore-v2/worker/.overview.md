# divinecore-v2/worker — Celery Worker + Beat

Background task execution and scheduled jobs. Same image runs both the worker process and the Beat scheduler — `docker-compose.yml` overrides the command for `beat`.

The worker image is built with **context = repo root** so the Dockerfile can copy in domain modules (`sales_os/`, `ops_os/`) alongside the local runtime files. Domain tasks live in those module folders; `celery_app.py` lists their import paths in `include=[...]`.

## Files

| File | Role |
|------|------|
| `celery_app.py` | Celery app config + `beat_schedule` + `include` list pointing into module integrations |
| `tasks.py` | Generic infra tasks (echo, heartbeat). Domain tasks live in `<module>/integrations/<vendor>/processor.py`. |
| `settings.py` | Pydantic Settings — currently shared across all integrations (Redis, OpenRouter, Fathom, Upwork). |
| `team.py` | TEAM_MEMBERS dict — used by Fathom for opt-in lookups. |
| `requirements.txt` | Python deps (celery, redis, googleapiclient, openai, supabase, etc.) |
| `Dockerfile` | Container build (context = repo root) |

## Beat schedule

In `celery_app.py`:

```python
app.conf.beat_schedule = {
    "heartbeat-every-30s": {"task": "tasks.heartbeat", "schedule": 30.0},
}
```

## Adding a new task

1. **Domain task**: add it to the right module — `<module>/integrations/<vendor>/processor.py`. Decorate with `@app.task(name="tasks.<name>")`. Add the module path to `include=[...]` in `celery_app.py`. Update the worker Dockerfile if the module folder isn't yet copied in.
2. **Generic infra task**: add it to `tasks.py` here.
3. **Use the `tasks.*` namespace** — the api service calls tasks by string name and expects this namespace.
4. If it should run on a schedule, add an entry to `app.conf.beat_schedule` in `celery_app.py`.
5. Rebuild the worker image (CI handles this on push to main).

## Notes

- `celerybeat-schedule` is the on-disk schedule state file produced by Beat; ignore it (and don't commit it).
- `__pycache__/` is gitignored.
- Tasks run inside the worker container — they have no access to local filesystem state outside the mounted `./worker` volume.

## What does NOT go here

- HTTP routes — those live in `api/main.py`.
- Long-lived service code (websocket listeners, polling loops as their own services) — make a new top-level folder and add a service to `docker-compose.yml`.
