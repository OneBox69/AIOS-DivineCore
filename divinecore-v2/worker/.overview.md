# divinecore-v2/worker — Celery Worker + Beat

Background task execution and scheduled jobs. Same image runs both the worker process and the Beat scheduler — `docker-compose.yml` overrides the command for `beat`.

## Files

| File | Role |
|------|------|
| `celery_app.py` | Celery app config + `beat_schedule` |
| `tasks.py` | Task implementations (registered via `@app.task(name=...)`) |
| `requirements.txt` | Python deps (celery, redis, etc.) |
| `Dockerfile` | Container build |

## Tasks today

| Name | Function | Purpose |
|------|----------|---------|
| `tasks.echo` | `echo(message)` | Demo — uppercases the input. Used by `POST /tasks/echo`. |
| `tasks.heartbeat` | `heartbeat()` | Demo scheduled task — logs UTC timestamp every 30s via Beat. |

## Beat schedule

In `celery_app.py`:

```python
app.conf.beat_schedule = {
    "heartbeat-every-30s": {"task": "tasks.heartbeat", "schedule": 30.0},
}
```

## Adding a new task

1. Add a function to `tasks.py` decorated with `@app.task(name="tasks.<name>")`.
2. **Use the `tasks.*` namespace** — the api service calls tasks by string name and expects this namespace.
3. If it should run on a schedule, add an entry to `app.conf.beat_schedule` in `celery_app.py`.
4. Rebuild the worker image (CI handles this on push to main).

## Notes

- `celerybeat-schedule` is the on-disk schedule state file produced by Beat; ignore it (and don't commit it).
- `__pycache__/` is gitignored.
- Tasks run inside the worker container — they have no access to local filesystem state outside the mounted `./worker` volume.

## What does NOT go here

- HTTP routes — those live in `api/main.py`.
- Long-lived service code (websocket listeners, polling loops as their own services) — make a new top-level folder and add a service to `docker-compose.yml`.
