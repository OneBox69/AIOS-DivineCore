# worker/integrations/fathom/

Pulls meeting recordings from Fathom's REST API on a schedule, categorizes them, and writes them to Supabase Postgres. Action items assigned to opted-in team members become rows in a separate `tasks` table. Fully outbound — no public ingress required.

## Flow

```
Beat (every 10 min)
  -> tasks.poll_fathom_recordings              # poller.py
       -> read last-polled checkpoint from Redis (key: fathom:last_polled_at)
       -> GET https://api.fathom.ai/external/v1/meetings
            ?created_after=<checkpoint>
            &include_summary=true
            &include_transcript=true
            &include_action_items=true
            &cursor=...                        # paginates until exhausted
       -> for each new meeting:
            tasks.process_fathom_recording.delay(item)   # processor.py
              -> categorizer.classify(item)              # heuristic + OpenRouter fallback
              -> supabase_writer.upsert(item, category)  # idempotent on meeting_id
              -> tasks_writer.create_tasks_for_opted_in(item, meeting_record)
       -> save new checkpoint timestamp to Redis
```

## Files

| File | Role |
|------|------|
| `poller.py` | Celery beat task. Pulls new meetings since last checkpoint, fans out one `process_fathom_recording` task per meeting, advances the checkpoint. |
| `processor.py` | Per-meeting Celery task. Composes categorize -> upsert -> create-tasks. The tasks step is wrapped in try/except so a tasks-table failure doesn't lose the meeting. |
| `categorizer.py` | Two-stage classifier: heuristic on attendee composition, then OpenRouter (default `openai/gpt-4o-mini`) fallback. |
| `supabase_writer.py` | Upserts the `meetings` row using `on_conflict="meeting_id"`. Truncates transcripts >90k chars. |
| `tasks_writer.py` | Inserts rows into the `tasks` table for action items whose assignee is a team member with `auto_pulse_tasks: True`. Otherwise no-op. |

## Supabase schema

The user runs this once in the Supabase SQL editor before deploying:

```sql
create table if not exists public.meetings (
    meeting_id     text primary key,
    title          text,
    date           timestamptz,
    duration_min   int,
    owner          text,
    attendees      text,
    external_attendees text,
    category       text check (category in ('client','strategy','cofounder','sales','other')),
    summary        text,
    transcript     text,
    transcript_url text,
    recording_url  text,
    processed_at   timestamptz not null default now()
);
create index if not exists meetings_date_idx on public.meetings (date desc);
create index if not exists meetings_category_idx on public.meetings (category);

create table if not exists public.tasks (
    id           uuid primary key default gen_random_uuid(),
    description  text not null,
    assignee     text,
    meeting_id   text references public.meetings(meeting_id) on delete set null,
    category     text,
    status       text not null default 'in-progress' check (status in ('in-progress','done','skipped')),
    priority     text not null default 'medium' check (priority in ('low','medium','high')),
    deadline     timestamptz,
    created_by   text,
    notes        text,
    created_at   timestamptz not null default now()
);
create index if not exists tasks_assignee_idx on public.tasks (assignee);
create index if not exists tasks_status_idx on public.tasks (status);
create index if not exists tasks_meeting_id_idx on public.tasks (meeting_id);

alter table public.meetings enable row level security;
alter table public.tasks enable row level security;
```

RLS is enabled but no policies are defined — the worker uses the **secret key** (`sb_secret_*`) which bypasses RLS. The publishable key (`sb_publishable_*`) cannot read or write either table. This means the data is private by default; future read endpoints (e.g. a custom Pulse digest) need either the secret key or explicit RLS policies.

## Fathom API details

- Base URL: `https://api.fathom.ai/external/v1`
- Auth header: `X-Api-Key: <key>`
- Endpoint used: `GET /meetings` with `created_after`, `cursor`, `include_summary=true`, `include_transcript=true`, `include_action_items=true`
- Response shape: `{ limit, next_cursor, items: [...] }`. Each item carries `recording_id`, `meeting_title`, `recording_start_time`/`recording_end_time`, `recorded_by`, `calendar_invitees`, `default_summary`, `transcript`, `action_items`, `url`, `share_url`.

Real payload field shapes (all confirmed against live API):
- `default_summary` is `{template_name, markdown_formatted}` (we extract `markdown_formatted`)
- `transcript` is a list of `{speaker: {display_name}, text, timestamp}` segments (we stringify into `[ts] name: text` lines)
- `action_items[].assignee` is `{name, email, team}` — email is often null, so we fall back to name

## Env vars

Read from `worker/settings.py` (Pydantic Settings):
- `FATHOM_API_KEY`, `FATHOM_API_BASE_URL` (default `https://api.fathom.ai/external/v1`), `FATHOM_POLL_LOOKBACK_HOURS` (default 24 — only matters on cold start when the Redis checkpoint is empty)
- `SUPABASE_URL`, `SUPABASE_SECRET_KEY`
- `OPENROUTER_API_KEY`, `OPENROUTER_BASE_URL` (default `https://openrouter.ai/api/v1`), `CATEGORIZER_MODEL` (default `openai/gpt-4o-mini`)

## Polling cadence + checkpoint

Beat schedule key: `poll-fathom-every-10m` (in `worker/celery_app.py`). Checkpoint timestamp persists in Redis at `fathom:last_polled_at`. On cold start (no checkpoint), poller looks back `FATHOM_POLL_LOOKBACK_HOURS` (default 24).

To force a re-pull of older meetings:
```bash
docker compose exec redis redis-cli SET fathom:last_polled_at '2026-05-01T00:00:00Z'
docker compose exec worker celery -A celery_app call tasks.poll_fathom_recordings
```

## Categorization

Heuristic order:
1. No external attendees + 2 internal -> `cofounder`
2. No external attendees + 3+ internal -> `strategy`
3. Any external attendee on `CLIENT_DOMAINS` -> `client`
4. Any external attendee on `SALES_PROSPECT_EMAILS` -> `sales`
5. Otherwise -> OpenRouter LLM fallback (returns one of the five categories or `other`)

Client/sales lists are constants in `categorizer.py`. Promote to a Supabase lookup table only when they grow past ~10 entries each.

## Why Supabase instead of Airtable

Originally written against Airtable. Switched after the workspace hit its monthly API request quota and started returning 429 PUBLIC_API_BILLING_LIMIT_EXCEEDED on every read/write. Supabase free tier has no per-request quota (only a 500MB DB cap), which fits this volume comfortably.

## Why no Pulse bridge

Earlier design wrote action items into Pulse's existing Airtable `Task` table so the Pulse n8n bot would pick them up. That bridge was removed when the user decided to rebuild Pulse per-person rather than use the shared one. Action items now go to Supabase `tasks` only — Pulse stays exactly as it is, untouched. Future Pulse rebuilds can read from `tasks` directly.
