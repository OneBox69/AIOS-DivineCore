# worker/integrations/fathom/

Processes Fathom webhook payloads forwarded by `api/routes/fathom.py`. Single Celery task entry point: `tasks.process_fathom_recording`.

## Flow

```
process_fathom_recording(payload)
  -> categorizer.classify(payload)              # heuristic, then Claude fallback
  -> airtable_writer.upsert(payload, category)  # idempotent on Meeting ID
  -> pulse_bridge.create_tasks_for_opted_in(payload, meeting_record)
```

## Files

| File | Role |
|------|------|
| `processor.py` | Celery task entry point. Composes the three steps. |
| `categorizer.py` | Two-stage classifier. Heuristic first (attendee composition + client/sales lists), OpenRouter LLM fallback (default `openai/gpt-4o-mini`) if heuristic returns nothing. |
| `airtable_writer.py` | Upserts the `Meetings` row. Looks up by `Meeting ID` first; updates if found, creates if not. Truncates transcripts >90k chars. |
| `pulse_bridge.py` | Creates rows in Pulse's existing `Task` table for action items whose assignee is a team member with `auto_pulse_tasks: True`. Otherwise no-op. |

## Airtable schema (Meetings table, DivineCore base)

`Meeting ID` (primary), `Title`, `Date`, `Duration (min)`, `Owner`, `Attendees`, `External Attendees`, `Category` (single-select: client/strategy/cofounder/sales/other), `Summary`, `Transcript`, `Transcript URL`, `Recording URL`, `Processed At`.

Action items are NOT stored on the Meetings row. They're either turned into Pulse Task rows (for opted-in users) or left in Fathom (source of truth via `Recording URL`).

## Env vars

Read from `worker/settings.py` (Pydantic Settings):
- `AIRTABLE_API_KEY`, `AIRTABLE_BASE_ID`
- `AIRTABLE_MEETINGS_TABLE` (default `Meetings`), `AIRTABLE_TASKS_TABLE` (default `Task`)
- `OPENROUTER_API_KEY`, `OPENROUTER_BASE_URL` (default `https://openrouter.ai/api/v1`), `CATEGORIZER_MODEL` (default `openai/gpt-4o-mini` — same model Pulse uses)

## Webhook signature

HMAC-SHA256 over the raw body, header `X-Fathom-Signature`. **Confirm the exact header name + algorithm against Fathom's docs once a real account is connected** — implementation is a reasonable default but unverified.

## Categorization

Heuristic order:
1. No external attendees + 2 internal -> `cofounder`
2. No external attendees + 3+ internal -> `strategy`
3. Any external attendee on `CLIENT_DOMAINS` -> `client`
4. Any external attendee on `SALES_PROSPECT_EMAILS` -> `sales`
5. Otherwise -> OpenRouter LLM fallback (returns one of the five categories or `other`)

Client/sales lists are constants in `categorizer.py`. Promote to an Airtable lookup table only when they grow past ~10 entries each.
