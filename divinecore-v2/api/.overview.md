# divinecore-v2/api — FastAPI Service

The HTTP entry point for the v2 runtime. Submits Celery tasks via Redis and lets callers poll for results.

The api image is built with **context = repo root** so the Dockerfile can copy in `sales_os/` (and any future module that exposes a router) alongside the local `api/` files. Module routers are imported in `main.py` and mounted via `app.include_router(...)`.

## Files

| File | Role |
|------|------|
| `main.py` | FastAPI app: inline `/` and `/tasks/*` routes + Celery client (`send_task` / `AsyncResult`); mounts routers imported from sibling modules (e.g. `from sales_os.web import upwork_router`) |
| `settings.py` | Pydantic Settings (REDIS_URL) |
| `requirements.txt` | Python deps (FastAPI, Celery, Jinja2, python-multipart for form parsing) |
| `Dockerfile` | Container build (context = repo root) |

Domain routes + templates mounted by this service:
- [`sales_os/web/upwork_routes.py`](../../sales_os/web/upwork_routes.py) → `/upwork`, `/upwork/finalize`

## Key contracts

- **Celery client** is a shared Celery app named `"api"` pointing at `REDIS_URL`. It calls tasks by string name (e.g. `"tasks.echo"`) — it does NOT import the worker code. This keeps api/ and worker/ independently buildable.
- **Task name discipline**: register all tasks under the `tasks.*` namespace. The api service only knows the string name.
- **Module routers** locate their own templates via `Path(__file__).resolve().parent / "templates"` so they work regardless of how / where the api is started.

## Endpoints

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/` | Health check — returns `{status, service}` |
| POST | `/tasks/echo` | Body `{"message": "..."}` — submits `tasks.echo`, returns `{task_id, submitted}` |
| GET | `/tasks/{task_id}` | Poll status + result from Celery's result backend |
| GET | `/upwork` | Renders the Upwork proposal generator form (sales_os/web) |
| POST | `/upwork` | Form-encoded `job_description` — submits `tasks.upwork_generate_proposal`, blocks on result |
| POST | `/upwork/finalize` | Form-encoded `row_index` + `base_connects` + `boosted_connects` + `loom_url` — submits `tasks.upwork_finalize_proposal` |

## Adding a new endpoint

1. **Domain endpoint**: add a router in `<module>/web/<name>_routes.py`, expose it from `<module>/web/__init__.py`, import + mount in `main.py`. Update the api Dockerfile if the module folder isn't yet copied in.
2. **Generic endpoint**: inline in `main.py`.
3. Make sure the matching Celery task exists with a `name="tasks.<name>"` decorator.
4. Add deps to `requirements.txt` if needed, then rebuild the api image.

## What does NOT go here

- Task implementations — those live in `<module>/integrations/<vendor>/processor.py` (or `worker/tasks.py` for generic infra tasks).
- Long-running compute — submit a Celery task, return the `task_id`, let the caller poll.
- Direct database writes from request handlers — go through a task.
