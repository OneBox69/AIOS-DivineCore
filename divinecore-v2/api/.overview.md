# divinecore-v2/api — FastAPI Service

The HTTP entry point for the v2 runtime. Submits Celery tasks via Redis and lets callers poll for results.

## Files

| File | Role |
|------|------|
| `main.py` | FastAPI app: routes + Celery client (`send_task` / `AsyncResult`) |
| `requirements.txt` | Python deps |
| `Dockerfile` | Container build |

## Key contracts

- **Celery client** is a shared Celery app named `"api"` pointing at `REDIS_URL`. It calls tasks by string name (e.g. `"tasks.echo"`) — it does NOT import the worker code. This keeps api/ and worker/ independently buildable.
- **Task name discipline**: register all tasks under the `tasks.*` namespace (see worker `tasks.py`). The api service only knows the string name.

## Endpoints

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/` | Health check — returns `{status, service}` |
| POST | `/tasks/echo` | Body `{"message": "..."}` — submits `tasks.echo`, returns `{task_id, submitted}` |
| GET | `/tasks/{task_id}` | Poll status + result from Celery's result backend |

## Adding a new endpoint

1. Define a Pydantic request model (or use a path/query param).
2. Add a route in `main.py` that calls `celery_client.send_task("tasks.<name>", args=[...])`.
3. Make sure the task exists in `worker/tasks.py` with a matching `name="tasks.<name>"` decorator.
4. Add the dep to `requirements.txt` if you pull in a new package, then rebuild the api image.

## What does NOT go here

- Task implementations — those live in `worker/tasks.py`.
- Long-running compute — submit a Celery task, return the `task_id`, let the caller poll.
- Direct database writes from request handlers (when DBs land) — go through a task or a thin service module.
