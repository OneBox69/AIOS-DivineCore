# divinecore-v2 — Code-First Runtime

The Python code-first runtime that runs in parallel to the n8n orchestration stack. Long-term, this replaces n8n for agent execution. Today, it's a foundation: API + worker + scheduler + broker, all containerised.

## Structure

```
divinecore-v2/
├── api/                       ← FastAPI service (.abstract.md + .overview.md here)
│   ├── main.py                ← Routes + Celery client
│   ├── requirements.txt
│   └── Dockerfile
├── worker/                    ← Celery worker + beat (.abstract.md + .overview.md here)
│   ├── celery_app.py          ← Celery config + beat_schedule
│   ├── tasks.py               ← echo, heartbeat
│   ├── requirements.txt
│   └── Dockerfile
├── docker-compose.yml         ← redis + api + worker + beat
└── .dockerignore
```

## Services (docker-compose.yml)

| Service | Image / Build | Notes |
|---------|---------------|-------|
| `redis` | `redis:7-alpine`, port 6379 | Broker AND result backend for Celery |
| `api` | `./api` | FastAPI on `:8000`. Mounts `./api` for live reload in dev. |
| `worker` | `./worker` | Celery worker. Same image as `beat`. |
| `beat` | `./worker` (override CMD) | Celery Beat scheduler. Currently runs `tasks.heartbeat` every 30s. |

`REDIS_URL=redis://redis:6379/0` is passed to all three Python services.

## Endpoints (today)

- `GET /` — health
- `POST /tasks/echo` — submit a Celery task, returns `task_id`
- `GET /tasks/{task_id}` — poll status + result

## Run locally

```bash
cd divinecore-v2
docker compose up --build
```

## What goes here

- New Celery tasks → `worker/tasks.py` (and registered in `celery_app.py` if needed)
- New API routes → `api/main.py`
- New scheduled jobs → add to `app.conf.beat_schedule` in `worker/celery_app.py`
- New service (e.g. a Discord listener) → new top-level folder + add to `docker-compose.yml`

## What does NOT go here

- n8n workflow JSONs — those go in the relevant module's `workflows/` folder.
- Agent system prompts — those go in the relevant module's `agents/` folder.
- Knowledge-base sync scripts — those go in the relevant module's `knowledge-base/` folder.
