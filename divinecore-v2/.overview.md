# divinecore-v2 — Code-First Runtime

The Python code-first runtime that runs in parallel to the n8n orchestration stack. Long-term, this replaces n8n for agent execution.

This folder hosts the **runtime only** — FastAPI app, Celery worker + beat, Redis broker, Docker Compose, env wiring. Domain code (Upwork, Fathom, future module integrations) lives in the module folders (`sales_os/`, `ops_os/`, etc.) and is imported by the runtime.

## Structure

```
divinecore-v2/
├── api/
│   ├── main.py                ← FastAPI app, mounts routers from sibling modules
│   ├── settings.py            ← Pydantic Settings (REDIS_URL)
│   ├── requirements.txt
│   └── Dockerfile             ← context = repo root; copies api/ + sales_os/
├── worker/
│   ├── celery_app.py          ← Celery config + beat_schedule + task includes (paths into sales_os/, ops_os/)
│   ├── tasks.py               ← echo, heartbeat (demo)
│   ├── settings.py            ← Pydantic Settings (Supabase, OpenRouter, Fathom, Google OAuth, Redis)
│   ├── team.py                ← TEAM_MEMBERS dict + email/name lookup
│   ├── requirements.txt
│   └── Dockerfile             ← context = repo root; copies worker/ + sales_os/ + ops_os/
├── docker-compose.yml         ← local dev (build context = .., volume mounts of module dirs, uvicorn --reload)
└── docker-compose.prod.yml    ← what's deployed on VPS (image: from GHCR, env_file: .env)
```

Domain code mounted into containers from outside this folder:
- `sales_os/integrations/upwork/` — Upwork pipeline (Celery tasks)
- `sales_os/web/upwork_routes.py` + templates — `/upwork` UI mounted into FastAPI
- `ops_os/integrations/fathom/` — Fathom poller + processor (Celery tasks)

## Services (docker-compose.yml)

| Service | Image / Build | Notes |
|---------|---------------|-------|
| `redis` | `redis:7-alpine`, port 6379 | Broker AND result backend for Celery |
| `api` | builds from `divinecore-v2/api/Dockerfile`, context = repo root | FastAPI on `:8000`. Volumes mount `./api` and `../sales_os` for hot reload. |
| `worker` | builds from `divinecore-v2/worker/Dockerfile`, context = repo root | Celery worker. Mounts `./worker`, `../sales_os`, `../ops_os`. |
| `beat` | same image as `worker` (override CMD) | Celery Beat scheduler. Runs `tasks.heartbeat` every 30s and `tasks.poll_fathom_recordings` every 10 min. |

`REDIS_URL=redis://redis:6379/0` is passed to all three Python services.

## Endpoints (today)

- `GET /` — health
- `POST /tasks/echo` — submit a Celery task, returns `task_id`
- `GET /tasks/{task_id}` — poll status + result
- `GET /upwork` — Upwork proposal generator form. See [sales_os/web/upwork_routes.py](../sales_os/web/upwork_routes.py).
- `POST /upwork` — runs the Upwork pipeline, blocks on result, renders application body + Doc URLs.
- `POST /upwork/finalize` — patches connects + Loom into the existing tracking-sheet row.

## Scheduled tasks

- `tasks.heartbeat` — every 30s (liveness)
- `tasks.poll_fathom_recordings` — every 10 min (Fathom integration; see [ops_os/integrations/fathom/.overview.md](../ops_os/integrations/fathom/.overview.md))

## Run locally

```bash
cd divinecore-v2
docker compose up --build
```

(Build context is `..` — the compose file resolves it; you don't need to cd up.)

## Deploy to VPS (manual, until CI is wired)

```bash
ssh root@srv1445995.hstgr.cloud \
  "cd /root/divinecore-v2 && \
   docker compose -f docker-compose.prod.yml pull && \
   docker compose -f docker-compose.prod.yml up -d"
```

The compose file in `/root/divinecore-v2/` should mirror `divinecore-v2/docker-compose.prod.yml` in this repo. If you change one, sync the other.

## What goes here

- FastAPI app entry + generic settings → `api/main.py`, `api/settings.py`
- Celery wiring + scheduled-job declarations → `worker/celery_app.py`
- Generic infra tasks (echo, heartbeat) → `worker/tasks.py`
- Cross-module worker settings (env vars used by multiple integrations) → `worker/settings.py`
- Compose / Dockerfile changes → here

## What does NOT go here

- **Domain integrations** (Upwork, Fathom, future Slack/HubSpot/...) — those live in `<module>/integrations/<vendor>/`.
- **HTTP routes for a domain UI** — those live in `<module>/web/` and get imported into `api/main.py`.
- n8n workflow JSONs — module's `workflows/` folder.
- Agent system prompts — module's `agents/` folder.
- Knowledge-base sync scripts — module's `knowledge-base/` folder.
