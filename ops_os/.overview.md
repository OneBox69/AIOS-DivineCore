# Ops OS — Module 3 (Execution System)

The manual-automation layer. Highest-leverage function: real-time gap detection. When a pipeline reaches a step that needs human intervention (cost too high, judgment required, infra not built), Ops OS detects it and escalates to the right person.

## Status

**Planning** for the agent layer (Task / Reminder / Monitoring).

**Live**: `integrations/fathom/` — beat-driven poller that ingests Fathom meeting recordings, writes them to Supabase, and creates Pulse tasks for action items assigned to opted-in team members. Runs as Celery tasks loaded by `divinecore-v2/worker/celery_app.py`.

## Folder structure

```
ops_os/
├── agents/         ← System prompts for Task/Reminder/Monitoring agents (scaffold)
├── workflows/      ← n8n workflow exports (scaffold)
└── integrations/   ← External-system code (Fathom live)
    └── fathom/     ← Meeting poller + categorizer + task writer — see .overview.md inside
```

## What goes here

- Agent system prompts → `agents/` (one file per agent)
- n8n workflow JSON exports → `workflows/`
- External-system Python integrations (Celery tasks + helpers) → `integrations/<vendor>/`

## What does NOT go here

- Pulse task records themselves — those live in Airtable / Supabase, not files.
- Brand voice or persona content — `shared/context/`. Don't duplicate.

## Why Fathom is here (not in `pulse/` or `sales_os/`)

Fathom records sales calls and the meetings table is essentially a call log — there's a fair argument for `sales_os/`. The deciding factor: the integration's *job* is task escalation (action items → Pulse tasks for opted-in members), which is Ops OS's core function. The meeting-storage side is a useful by-product, not the reason the integration exists.
