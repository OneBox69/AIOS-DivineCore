# branding_os/integrations/social_perf/

Daily LinkedIn + X post tracker. Pulls posts off configured profiles via Apify once a day, stores them in Supabase with metrics, runs each post through an LLM to extract hook/closing/topic/format, and exposes a dashboard at `GET /social-perf` that surfaces which patterns correlate with engagement.

**Scaffolding status.** Folder structure, Celery tasks, Supabase schema, and dashboard are in place. Apify actor IDs (`apimaestro/linkedin-profile-posts`, `apidojo/tweet-scraper`) are reasonable defaults but unverified — expect to swap them and tighten the `_normalise_linkedin` / `_normalise_x` field mappings after the first real run, using the `raw` jsonb column as the source of truth.

## Why Apify (and not the official APIs)

LinkedIn's personal-post analytics aren't available via their public API — the Marketing Developer Platform is organization-scoped and gated by app approval. X's official API is $100/mo for the Basic tier and rate-limits aggressively. Apify actors scrape both, rotate their own proxies, and cost roughly $5–15/mo at one-run-per-day volume across both platforms. When the scrapers break or change shape, swap the actor ID via env and adjust the normaliser — everything downstream is shape-agnostic via the `raw` jsonb column.

## Flow

```
Beat (daily 22:10 UTC)
  -> tasks.poll_social_posts                                # processor.py
       for each url in SOCIAL_PERF_LINKEDIN_PROFILES.split(','):
         -> apify_client.fetch_linkedin_posts(url)
              -> POST https://api.apify.com/v2/acts/{linkedin_actor}/run-sync-get-dataset-items
         -> for each item: _normalise_linkedin -> supabase_writer.upsert_post
       for each handle in SOCIAL_PERF_X_HANDLES.split(','):
         -> apify_client.fetch_x_posts(handle)
              -> POST https://api.apify.com/v2/acts/{x_actor}/run-sync-get-dataset-items
         -> for each item: _normalise_x -> supabase_writer.upsert_post
       -> for each unannotated post in Supabase:
            -> send_task("tasks.annotate_social_post", [platform, post_id])

tasks.annotate_social_post (one Celery task per post, parallel)
  -> annotator.annotate(content)
       -> POST OpenRouter /chat/completions (JSON mode)
       -> { hook, closing, topic, format }
  -> supabase_writer.set_annotation(...)                    # patches just the annotation columns

Operator (any time)
  -> GET /social-perf?tab=all|linkedin|x
       -> supabase_writer.fetch_posts(platform=...)
       -> rank by engagement = likes + comments + reposts
       -> bucket stats by format + by topic
       -> rendered via social_perf_dashboard.html
```

## Files

| File | Role |
|------|------|
| `apify_client.py` | httpx wrapper around `POST /v2/acts/{id}/run-sync-get-dataset-items`. Two helpers: `fetch_linkedin_posts(profile_url)` + `fetch_x_posts(handle)`. 5-minute timeout. |
| `supabase_writer.py` | `upsert_post` (excludes annotation columns via `?columns=`), `fetch_unannotated`, `fetch_posts`, `set_annotation`. Composite primary key `(platform, post_id)`. |
| `annotator.py` | One OpenRouter call → `{ hook, closing, topic, format }`. JSON-mode response, format clamped to a fixed enum. Raises `AnnotationError` on missing config or malformed output. |
| `processor.py` | Three Celery tasks: `poll_social_posts` (beat), `annotate_social_post` (per-post fanout), `annotate_backfill` (manual one-off after prompt/model changes). All no-op cleanly on missing config. |

## Supabase schema

Run once in the Supabase SQL editor before deploying:

```sql
create table if not exists public.social_posts (
    platform           text not null check (platform in ('linkedin','x')),
    post_id            text not null,
    author_handle      text not null,
    url                text not null,
    posted_at          timestamptz,
    content            text not null,
    media_type         text,                                  -- e.g. image / video / article / text
    metrics            jsonb not null default '{}'::jsonb,    -- {likes, comments, reposts, impressions, bookmarks, ...}

    -- LLM-annotated (null until tasks.annotate_social_post runs).
    hook               text,
    closing            text,
    topic              text,
    format             text,                                  -- list / story / insight / question / case-study / hot-take / tutorial / announcement / other
    annotated_at       timestamptz,
    annotator_model    text,

    -- Provenance
    raw                jsonb not null,
    fetched_at         timestamptz not null default now(),
    metrics_updated_at timestamptz not null default now(),

    primary key (platform, post_id)
);

create index if not exists social_posts_platform_posted_idx
    on public.social_posts (platform, posted_at desc nulls last);
create index if not exists social_posts_format_idx
    on public.social_posts (format);
create index if not exists social_posts_topic_idx
    on public.social_posts (topic);
create index if not exists social_posts_unannotated_idx
    on public.social_posts (platform, post_id) where hook is null;

alter table public.social_posts enable row level security;
-- No policies — worker uses SUPABASE_SECRET_KEY which bypasses RLS.
```

## Annotator-owned columns

`hook`, `closing`, `topic`, `format`, `annotated_at`, `annotator_model` are owned by `tasks.annotate_social_post`, never by the daily poller. The poller's upsert uses PostgREST's `?columns=...` to whitelist only the polling fields — so a post that's been annotated and reappears in tomorrow's scrape gets its `metrics` refreshed without wiping the LLM annotations.

## Re-annotating after prompt/model changes

If you change `ANNOTATOR_SYSTEM_PROMPT` or bump `SOCIAL_PERF_ANNOTATOR_MODEL`, the existing rows will still hold the old annotation. To re-run on everything:

```sql
update public.social_posts set hook = null, closing = null, topic = null, format = null,
                               annotated_at = null, annotator_model = null;
```

Then trigger `tasks.annotate_backfill` (manually or just wait for the next nightly poll, which fans out annotation for everything with a null hook):

```bash
docker compose exec worker celery -A celery_app call tasks.annotate_backfill
```

## Apify setup (one-time)

1. Sign up at apify.com → Settings → Integrations → API token. Copy.
2. Add to `.env`: `APIFY_API_TOKEN=apify_api_...` (this is the same token already used by Upwork-jobs).
3. Defaults: `apimaestro/linkedin-profile-posts` + `apidojo/tweet-scraper`. Both are pay-per-result actors on the Apify Store; budget ~$5–15/mo for daily polling of a handful of profiles.

## Env vars

Read from `worker/settings.py`:
- `APIFY_API_TOKEN` — shared with Upwork-jobs
- `SOCIAL_PERF_LINKEDIN_PROFILES` — CSV of full profile URLs, empty = skip LinkedIn
- `SOCIAL_PERF_X_HANDLES` — CSV of bare handles (no `@`), empty = skip X
- `APIFY_LINKEDIN_ACTOR_ID`, `APIFY_X_ACTOR_ID` — actor overrides
- `SOCIAL_PERF_ANNOTATOR_MODEL` — OpenRouter model id, default `openai/gpt-4o-mini`
- `OPENROUTER_API_KEY`, `OPENROUTER_BASE_URL` — already configured (shared with Fathom + Upwork)
- `SUPABASE_URL`, `SUPABASE_SECRET_KEY` — already configured

Read from `api/settings.py`:
- `SUPABASE_URL`, `SUPABASE_SECRET_KEY` — read by the dashboard

## Manual trigger / debugging

Force the daily poll outside of beat:
```bash
docker compose exec worker celery -A celery_app call tasks.poll_social_posts
```

Re-run annotation for everything missing a hook:
```bash
docker compose exec worker celery -A celery_app call tasks.annotate_backfill
```

Inspect what Apify returned for a single post:
```sql
select platform, post_id, raw -> 'reactionsCount', raw -> 'commentsCount'
from public.social_posts
where platform = 'linkedin' and post_id = '<urn>';
```

## Open questions

1. **Whose profiles?** Mayank, Pang, or both — currently driven by env CSV so adding/removing an author is just an env change.
2. **Engagement metric.** Currently `likes + comments + reposts`. Impressions would be a better denominator (per-impression engagement rate) but most scrapers can't see impressions reliably — revisit once we have a few weeks of real data.
3. **Hook clustering / suggestion loop.** Step 2 of "what's working" is feeding the best-performing hooks back into a draft suggester. Not built yet — the data shape supports it; build it once the annotation pipeline has run for a couple of weeks.
4. **Apify field-name verification.** The `_normalise_linkedin` / `_normalise_x` mappings are best-effort fallbacks across several common actor shapes. After the first real run, inspect a few rows in Supabase Studio and tighten the mappings.
